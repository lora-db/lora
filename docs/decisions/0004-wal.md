# 0004 — Write-ahead log

Status: **Accepted**, shipped in v0.3.x.

## Context

LoraDB's open-source core was, before this decision, an in-memory graph
with on-demand snapshots ([0003-snapshot-format](0003-snapshot-format.md))
as its only durability mechanism. That gave operators a checkpoint they
could schedule, but every byte of data between two snapshot saves was
lost on crash — the open issue tracked in [known-risks.md](../design/known-risks.md)
under "No continuous persistence".

A recurring request from operators evaluating LoraDB for write-heavy
workloads was a continuous-durability story strong enough to recover
from crash without losing the last few minutes of writes. We had two
seams already in place:

- `MutationEvent` — the per-mutation vocabulary in `lora-store` ([snapshots.md](../operations/snapshots.md#mutation-events)),
  designed exactly for this use case.
- `MutationRecorder` — the optional observer hook on `InMemoryGraph`
  that fires on every committed mutation; default `None` so non-WAL
  builds pay only a null-pointer check per mutation.
- `wal_lsn` reserved field in the snapshot header — a fence position
  the WAL would refer to.

The decision was *how* to cash that hand in.

## Decision

Add a new `lora-wal` crate that:

- Owns the durable byte stream — segment files, framing, replay.
- Implements `MutationRecorder` via a `WalRecorder` adapter, so the
  storage trait does not need to learn about durability.
- Brackets every query with `arm → (lazy begin) → commit/abort → flush`
  while the engine mutex is held, so the on-disk WAL order equals the
  in-memory commit order with no extra synchronisation.

Wire it into `lora-database::Database` through two new constructors —
`Database::open_with_wal(WalConfig)` and `Database::recover(snapshot,
WalConfig)` — and through `Database::checkpoint_to(path)` which
emits a snapshot stamped with the current `durable_lsn`, appends a
`Checkpoint` record, and truncates the log up to that fence.

### Scope this PR ships

- **Rust crate API** (`lora-database` re-exports `WalConfig`, `SyncMode`).
- **`lora-server` HTTP transport** with the `--wal-dir` /
  `--wal-sync-mode` flags and the admin routes `/admin/wal/status`,
  `/admin/wal/truncate`, `/admin/checkpoint`.
- **Recovery from crash** — committed transactions on disk are
  replayed into a fresh graph on boot. Aborted (uncommitted-at-EOF)
  transactions are dropped.
- **Torn-tail recovery** — a CRC-failed record at the end of the
  active segment is truncated cleanly so subsequent appends pick up
  at the right boundary.
- **Three sync modes** — `PerCommit` (default), `Group { interval_ms }`
  (background flusher), `None` (no `fsync`). Background-flusher fsync
  failures latch the WAL into a poisoned state, surfaced via
  `/admin/wal/status` as `bgFailure`.
- **Checkpoint hybrid** — `Database::checkpoint_to` writes a snapshot
  with `wal_lsn` set to the durable LSN, appends a `Checkpoint`
  marker, and drops sealed segments at or below the fence.

### Scope this PR explicitly does **not** ship

- **Bindings parity.** Python, Node.js, Ruby, WASM, and Go FFI do
  not expose WAL configuration. They continue to use snapshots only.
  Adding a half-wired binding surface (e.g. `walDir` accepted but
  no admin parity) would be operator-confusing; the right move is
  the Rust + HTTP server today, and a separate decision when binding
  parity catches up.
- **Concurrent-with-queries truncation.** Segment deletion runs under
  the engine mutex. A long-running checkpoint blocks queries; this is
  acceptable today because the engine is mutex-serialised anyway and
  truncation is fast (a few `unlink` calls).
- **Multi-process exclusion.** Two processes with the same `--wal-dir`
  is undefined behaviour. Lock-file based exclusion is deferred.
- **Marker-driven recovery override.** `replay_dir` reports the
  newest checkpoint marker observed (`checkpoint_lsn_observed`), and
  `Database::recover` prints a stderr warning when the snapshot the
  operator passed is older than that marker. The marker does **not**
  override the snapshot's `wal_lsn` as the recovery fence — the
  conservative-correct behaviour is to replay every record above the
  loaded snapshot's fence, which is what the current code does.

## Trade-offs

- **`PerCommit` adds an `fsync` per write.** On NVMe that's 50–200 µs;
  on slow disks, hundreds of milliseconds. Operators who can tolerate
  ~50 ms of post-crash data loss can drop to `Group`.
- **Group-mode failure visibility.** Background fsync errors used to
  be swallowed silently. They now latch onto the WAL itself so the
  next foreground `commit` / `flush` returns `WalError::Poisoned`,
  which the recorder propagates to the host as a durability error.
  `/admin/wal/status` includes `bgFailure` for monitoring.
- **Read-only queries.** They cost zero WAL traffic. The recorder is
  *armed* (a flag flip) on every query; `Wal::begin` is deferred to
  the first mutation event. Pure reads therefore allocate no LSN,
  append no records, and skip the surrounding `flush()` entirely.
- **Snapshot rebuild on boot.** Recovery time is dominated by replay
  size. `wal_benchmarks` reports replay throughput on a fixed
  configuration; checkpoints bound the worst case.
