# ISO/IEC 39075:2024 GQL Reference

Created: 2026-05-03

This document is a practical, implementation-oriented brief for moving LoraDB
toward GQL, the ISO standard Graph Query Language. It is not a copy of the
standard text. The full ISO/IEC 39075:2024 document is copyrighted and sold by
ISO; this file summarizes public information and maps the relevant feature
groups against the features LoraDB already supports.

Primary public sources:

- [ISO/IEC 39075:2024 - Information technology - Database languages - GQL](https://www.iso.org/standard/76120.html)
- [Public ISO/IEC preview PDF](https://www.en-standard.eu/publicdoc/iec_previews/3543528.pdf)
- [GQL Standards: What is a GQL Standard?](https://www.gqlstandards.org/what-is-a-gql-standard)
- LoraDB source of truth: [`cypher-support-matrix.md`](./cypher-support-matrix.md)

## Definition

ISO/IEC 39075:2024 defines GQL, a database language for property graphs. The
official ISO abstract describes it as covering data structures and basic
operations on property graphs, including capabilities for creating, accessing,
querying, maintaining, and controlling graphs and their data. The public preview
also frames GQL as a declarative, transactional language for modeling,
querying, and modifying property graph data.

In practical engine terms, GQL standardizes four layers:

1. A property graph data model.
2. Graph pattern matching.
3. Query, update, and graph management statements.
4. Types, expressions, functions, transactions, diagnostics, sessions, and
   conformance rules.

GQL targets property graphs rather than RDF graphs. A property graph is made of
nodes, edges, labels, and properties. Nodes are also commonly called vertices;
edges are also commonly called relationships. A path is an ordered traversal
through nodes and edges.

## Relationship to Cypher

GQL adopted much of Cypher's query shape, especially the familiar
`MATCH ... RETURN` pattern. That makes LoraDB's current Cypher-like pipeline a
reasonable starting point for GQL support. The migration should not rewrite the
store or executor first. It should introduce a GQL parser and conformance layer
that lower into LoraDB's existing canonical query pipeline.

Current LoraDB pipeline:

```text
query string
  -> lora-parser
  -> lora-ast
  -> lora-analyzer
  -> lora-compiler
  -> lora-executor
  -> lora-store
```

Recommended future pipeline:

```text
query string
  -> language facade
      -> GQL parser
      -> Cypher compatibility parser
  -> canonical Lora query AST
  -> analyzer
  -> compiler
  -> executor
  -> store
```

## Conformance Vocabulary

Use these labels in LoraDB GQL planning documents:

| Status | Meaning |
|---|---|
| `Supported` | LoraDB has tested executable behavior today, although syntax may currently be Cypher-shaped. |
| `Partial` | LoraDB has adjacent support but needs syntax, semantic, or conformance work. |
| `Not implemented` | No parser, analyzer, compiler, or executor path exists today. |
| `Out of scope for MVP` | Standard area exists, but it is not needed for the first embedded GQL profile. |
| `Cypher compat only` | Keep only in the compatibility parser unless intentionally standardized in the GQL frontend. |

## GQL Feature Groups

The table below is organized around the public ISO summary, GQL standards
materials, and public conformance tables. It is intentionally an
engineering inventory, not a clause-by-clause reproduction of the ISO text.

| Feature group | GQL intent | LoraDB status | Notes |
|---|---|---|---|
| Property graph model | Graphs contain nodes, edges, labels, properties, and paths. | Supported | LoraDB already stores nodes, relationships, properties, labels, and path values. |
| Open graph type | Minimum conformance can support an open graph type. | Supported | LoraDB is schema-free today, which maps naturally to an open graph. |
| Graph pattern matching | Match graph structures through node and edge patterns. | Supported | Current `MATCH` implementation covers node, relationship, label, property, and multi-part patterns. |
| Declarative query composition | Compose reads, filters, projections, ordering, and pagination. | Supported | Current Cypher-like surface has `MATCH`, `WHERE`, `RETURN`, `WITH`, `ORDER BY`, `SKIP`, and `LIMIT`. |
| Graph updates | Insert/create, set/update, remove, and delete graph data. | Partial | LoraDB supports write operations, but GQL syntax and exact semantics still need a GQL frontend pass. |
| Graph management | Create/drop/use graphs and graph collections. | Not implemented | LoraDB has database open/clear APIs but no GQL graph catalog language yet. |
| Transactions | Commit and rollback graph work. | Partial | LoraDB has explicit transactions internally and through bindings, but no GQL transaction statement syntax. |
| Session and client model | User/session context and standard client interaction model. | Out of scope for MVP | Current embedded API and HTTP server can expose a narrower profile first. |
| Security model | Users, privileges, and access control. | Not implemented | No authentication or authorization language exists today. |
| Status objects | Standard diagnostics for success, warnings, errors, and exceptions. | Partial | LoraDB has parse/analyze/execute errors with spans, but not GQL-status objects. |
| Mandatory scalar property types | Boolean, integer, float, and string. | Supported | Current value model supports all four. |
| Null / unknown semantics | Three-valued logic and null behavior. | Partial | LoraDB has broad null behavior, but exact GQL aggregate/null edge cases need conformance tests. |
| Structural graph types | Node/vertex, edge/relationship, path. | Supported | Current result model includes node, relationship, and path values. |
| Lists and maps | Composite values and property maps. | Supported | LoraDB supports heterogeneous nested lists/maps in query values; property storage has stricter conversion rules. |
| Temporal values | Date/time/datetime/duration families. | Supported | Current engine supports date, time, local time, datetime, local datetime, and duration. |
| Spatial values | Point values and distance operations. | Supported | LoraDB supports Cartesian and WGS-84 points, including 3D point records. |
| Vector values | Vector data and vector functions. | Supported as extension | Useful LoraDB extension, not a baseline GQL feature group from public conformance tables. |
| Aggregation | Count, sum, avg, min, max, grouping. | Supported | Current engine also supports `collect`, standard deviations, and percentiles. |
| Path search modes | Shortest paths and path modes. | Partial | LoraDB supports `shortestPath()` and `allShortestPaths()` in Cypher form; GQL path search syntax differs. |
| Quantified paths | Standard quantified path and edge patterns. | Not implemented | Existing variable-length relationship syntax is Cypher-like, not GQL quantified paths. |
| Procedures / calls | Invoke graph procedures or functions. | Not implemented | Parser accepts `CALL`, analyzer rejects it as unsupported. |
| Normalized predicates | Unicode normalization predicates. | Partial | Scalar `string.normalize` supports NFC/NFD/NFKC/NFKD; dedicated GQL normalized predicates are not implemented. |
| Unicode identifiers and strings | Standard Unicode parsing behavior. | Partial | LoraDB stores UTF-8 strings; parser/conformance around Unicode identifiers needs a dedicated audit. |

## GQL MVP for LoraDB

The first strict GQL profile should be deliberately small and executable:

1. Parse GQL read queries for the existing property graph model.
2. Lower GQL node/edge patterns into the current AST/resolved pattern types.
3. Support `MATCH`, `WHERE`, `RETURN`, `ORDER BY`, `LIMIT`, and `OFFSET`.
4. Support mandatory scalar property types: boolean, integer, float, string.
5. Support nulls, property references, parameters, arithmetic, comparison,
   boolean logic, `CASE`, and aggregate functions.
6. Support open graph type semantics only.
7. Emit LoraDB's existing errors first, then add GQL-status object mapping.

This MVP avoids graph catalog DDL, security, sessions, procedure calls,
advanced path modes, and full transaction language until the parser facade and
core conformance tests are stable.

## Parser and API Migration Notes

Suggested code-level changes:

1. Add `QueryDialect`:

   ```rust
   pub enum QueryDialect {
       Gql,
       CypherCompat,
   }
   ```

2. Replace bare `parse_query(query)` calls with:

   ```rust
   parse_query_with_options(query, ParseOptions {
       dialect: QueryDialect::Gql,
   })
   ```

3. Key the plan cache by dialect plus query text, not raw text alone.

4. Keep the existing parser as `CypherCompat` first. Do not delete it during
   the first GQL workstream.

5. Introduce a new GQL grammar entry point and lower it into the same canonical
   AST used by the analyzer.

6. Add API-level dialect selection:

   ```rust
   db.execute_gql(query, options)
   db.execute_cypher_compat(query, options)
   ```

7. Keep `execute(query, options)` as compatibility behavior until a breaking
   release can flip the default to GQL.

## Known Gaps Against GQL Direction

High-priority gaps:

- GQL parser entry point.
- Dialect-aware plan cache.
- GQL `OFFSET` syntax.
- GQL `FOR` syntax mapped to existing `UNWIND` execution.
- GQL path search syntax mapped to shortest-path machinery.
- Quantified path patterns.
- GQL diagnostics/status object mapping.
- Exact aggregate/null conformance tests.
- Unicode identifier and normalization conformance.

Medium-priority gaps:

- Graph catalog syntax.
- Transaction statements.
- Procedure/call support.
- Session/user expression support.
- Type predicates and richer type declarations.
- GQL graph type constraints.

Out of initial scope:

- Security/authorization language.
- Multi-graph catalog management.
- Full client/session model.
- Vendor-specific Cypher features.
- Vector standardization; keep vectors as a LoraDB extension.

## Compatibility Policy Recommendation

Use GQL as the primary language target and keep Cypher as a limited
compatibility dialect:

| Area | Recommendation |
|---|---|
| New syntax | Add to GQL first. |
| Existing Cypher syntax | Keep only where it maps cleanly to existing execution. |
| Cypher-only features | Freeze unless critical for users. |
| Docs | Rename user-facing query docs toward GQL once parser exists. |
| Tests | Keep old Cypher tests as compatibility fixtures; add GQL conformance fixtures beside them. |
| Error messages | Preserve friendly LoraDB messages, then attach GQL status codes. |

This lets the engine move toward ISO GQL without throwing away the working
Cypher-like parser or forcing all bindings to change in one release.
